# Simular Design

## Look And Feel

- Research-forward product page with benchmark cards, blog/research links, and minimal direct UI chrome in the public pages.
- The design leans frontier-lab more than productivity extension: benchmark percentages, research history, foundation-agent language, and platform breadth.
- Product copy frames the agent as inspectable and programmable through Simulang, a JavaScript-like DSL for precise repeatable automation.

## Design Tokens To Track

```yaml
surface: clean research/product hybrid
accent: benchmark percentages and platform badges
primary_control: natural-language task plus Simulang automation script
secondary_controls:
  - macOS download
  - Windows waitlist
  - web app
  - webhook integration
  - readable/modifiable execution
trust_controls:
  - transparent execution
  - inspectable automation
  - repeatable DSL
information_density: medium
```

## Differentiators

- Positions across browser, computer, and smartphone benchmarks rather than only browser automation.
- Uses a custom DSL to make agent actions readable, inspectable, and modifiable.
- Claims production-grade reliability for workflows with very long step counts.

## What Is Good

- Benchmark-led messaging communicates technical seriousness.
- Transparent execution is a strong design principle for trust in computer-use agents.
- Desktop-wide scope expands the addressable market beyond web forms and browser tabs.

## Where It Breaks Down

- Public pages show less concrete everyday workflow UX than extension/browser products, so buyers may need demos to understand the actual interaction model.
- Frontier benchmark copy can feel abstract compared with products that show live browser sessions, logs, and approval controls.
- Whole-computer automation increases governance burden because files, apps, OS permissions, and browser state are all in scope.
