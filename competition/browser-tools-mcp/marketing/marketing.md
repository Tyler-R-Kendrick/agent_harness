# BrowserTools MCP Marketing

## Audience

- AI-assisted frontend developers using Cursor, Cline, Zed, Claude Desktop, or similar MCP hosts.
- Teams debugging web apps where the agent needs screenshots, console logs, network traces, and selected DOM context.
- Developers who want local browser evidence without sending browser logs through a hosted browser platform.

## Positioning

BrowserTools MCP positions itself as a practical awareness layer for coding agents: the agent becomes more capable because it can inspect the same browser artifacts a developer would inspect in DevTools.

The strongest pitch is not autonomous browsing. It is faster UI debugging with the browser still local and visible.

## Customer Model

- Open-source distribution through GitHub and npm commands.
- Developer adoption via MCP-client setup examples and Cursor-oriented usage.
- Brand funnel for AgentDeskAI's broader developer-agent infrastructure.

## Who It Captures

- Developers who already trust MCP tools and want immediate browser diagnostics.
- Cursor-heavy teams that need UI bug evidence in the coding loop.
- Local-first users who prefer an extension and localhost server over a hosted browser farm.

## Who It Does Not Capture

- Enterprises that require an actively maintained vendor-backed product.
- Non-technical operators who cannot reason about DevTools, npm, and MCP config.
- Teams that need replayable browser sessions, policy approvals, or hosted scale rather than local evidence capture.

## Competitive Takeaway

BrowserTools MCP validates a core `agent-browser` wedge: browser-agent UX is stronger when screenshots, logs, network traces, selected DOM, and audit results are first-class run evidence.
