# Hyperbrowser Marketing

## Audience

- AI developers building browser agents, scraping products, QA automation, data extraction, and browser-backed tools.
- Teams that need managed browser fleets without operating Chrome infrastructure.
- Agent-framework users who want a hosted browser substrate for Browser Use, Claude Computer Use, OpenAI CUA, Gemini Computer Use, or custom Playwright code.

## Positioning

- Core claim: fast cloud browsers for AI agents.
- Market frame: AI products need the live web, but local browsers and ad hoc automation infrastructure do not scale cleanly.
- Product promise: launch cloud browser sessions, scrape or crawl pages, run AI agents, integrate MCP, and inspect recordings from one platform.

## Customer Model

- Credit-based SaaS usage model.
- Monetization units include browser-hours, proxy bandwidth, fetch/search/scrape pages, AI extraction output tokens, per-step agent execution, and model tokens.
- Open-source HyperAgent pulls developers into the platform while cloud sessions and dashboards drive hosted usage.

## Who They Capture

- Infrastructure-minded developers who already know Playwright/Puppeteer/Selenium.
- Startups building browser-agent products that need scale quickly.
- Teams that want a single vendor to host multiple browser-agent modes and web data APIs.

## Who They Miss

- End users who want an opinionated browser workspace rather than APIs.
- Teams that need local-only execution for sensitive browsing.
- Buyers who prefer fewer primitives and a narrower, more guided product path.

## Competitive Read

Hyperbrowser is a strong threat to the hosted execution layer around `agent-browser`. The product is less threatening to local-first UX, artifact curation inside a workspace, and human-visible agent governance, where `agent-browser` can feel more inspectable.
