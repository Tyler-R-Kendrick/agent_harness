# Browserbeam Marketing

## Audience

- AI-agent developers who want browser control without running Playwright infrastructure.
- Teams building form filling, web scraping, QA testing, screenshot/PDF, and LLM-powered browsing workflows.
- Builders who care about token cost and reliable page state more than full cloud-agent platform breadth.

## Positioning

- Core claim: the browser API built for AI agents.
- Market frame: existing browser clouds expose hosted Playwright or markdown extraction, but agents need structured interactive state.
- Product promise: create a session, observe structured JSON, interact using refs, receive diffs, and avoid managing browser infrastructure.

## Customer Model

- Self-serve SaaS with a free credit trial.
- Monthly plans scale by credit pool, concurrency, and session duration.
- Usage burns credits across runtime, proxy bandwidth, AI selectors, and CAPTCHA solves.

## Who They Capture

- Developer teams that want a direct HTTP API rather than a browser SDK.
- Agent builders looking to reduce prompt context with refs and diffs.
- Small teams comparing Browserbase, Browserless, Steel, and Firecrawl on implementation speed.

## Who They Miss

- Users who need a visible local browser workspace and manual handoff.
- Teams that need deep Playwright/CDP escape hatches.
- Buyers who want packaged workflow automation or human approval UX instead of a low-level API.
