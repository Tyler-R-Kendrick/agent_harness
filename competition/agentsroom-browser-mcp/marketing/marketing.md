# AgentsRoom Browser MCP Marketing

## Audience

- Developers using Claude Code, Codex CLI, Gemini CLI, OpenCode, or Aider.
- Teams running multiple coding agents in parallel across several local projects.
- Builders who want agents to verify actual UI behavior before declaring work complete.

## Positioning

- Core claim: all your agents in one screen.
- Browser-specific claim: your agents can click, type, and test a real browser.
- Market frame: terminal-tab sprawl makes parallel agents hard to manage, and coding agents often claim success without opening the page.
- Product promise: a local-first agent cockpit with roles, status, terminal control, browser automation, diffs, prompt library, backlog, mobile companion, and provider choice.

## Customer Model

- The observed site pushes free download and BYOK provider usage.
- It says AgentsRoom is not a token reseller and that user quotas stay with the provider.
- Pricing for the desktop product or browser feature was not clear in the public pages reviewed.

## Who They Capture

- Power users who already run several terminal agents and need visual organization.
- Developers who value QA-agent roles and visible browser verification.
- Teams who want local project folders and provider choice rather than a cloud-only coding-agent workspace.

## Who They Miss

- Users who only want a lightweight MCP server for browser automation.
- Organizations that need explicit pricing, admin controls, and procurement documentation up front.
- Non-coding automation teams whose browser tasks are not tied to local dev projects.
