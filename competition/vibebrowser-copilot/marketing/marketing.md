# VibeBrowser Co-Pilot Marketing

## Audience

- Developers using Codex, Claude Code, OpenCode, Gemini CLI, Cursor, or Copilot who need real browser control.
- Power users in recruiting, sales, research, legal, tax, crypto, and operations who work in logged-in web apps.
- Teams that want automation in Chrome without rebuilding workflows against APIs.

## Positioning

- Core claim: an agentic browser co-pilot that operates the web from the user's real browser session.
- Category frame: local-first browser agent plus MCP bridge, not a cloud-only browser farm.
- Trust frame: human-in-control guardrails, model choice, local sessions, and a secrets vault.

## Customer Model

- Public messaging emphasizes extension install and MCP setup; the open-source MCP package lowers adoption friction.
- Likely paid layers are remote relay, managed assistant/cloud-hosted OpenClaw-style workflows, team sharing, and enterprise controls.

## Who They Capture

- Agent-tooling users who already believe MCP is the right integration layer.
- Authenticated SaaS users who do not want separate browser profiles or cloud browser sessions.
- Users who prefer BYOK or local model routing over a closed AI browser subscription.

## Who They Miss

- Enterprise buyers who need a mature admin console, compliance proof, and procurement path.
- Nontechnical users who do not know what MCP, relay ports, or agent tools mean.
- Teams that prefer disposable isolated cloud browsers for security instead of granting access to a real personal browser.
