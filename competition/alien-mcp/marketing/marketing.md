# AlienMcp Marketing

## Audience

- Developers using Claude Code, Cursor, VS Code, or another MCP-compatible client.
- Local-first users who want agents to operate their existing authenticated Chrome tabs.
- Builders who prefer a small open-source bridge over a hosted browser platform.

## Positioning

- Core claim: connect any MCP client to real Chrome tabs with existing sessions, cookies, and extensions.
- Trust wedge: 100% local operation, no analytics, no tracking, and tab-group scoping.
- Technical wedge: 19 practical browser tools and trusted CDP events instead of headless browser automation.

## Customer Model

- MIT open source with no visible paid tier in the public page.
- Adoption likely depends on GitHub trust, community examples, and compatibility with popular MCP clients.

## Who They Capture

- Power users who already know MCP and want a quick bridge from coding agent to real browser.
- Developers who need to debug logged-in web apps with console/network visibility.
- Privacy-sensitive users who reject hosted browser sessions.

## Who They Miss

- Non-technical users who need one-click install and guided approvals.
- Teams that require centralized policy, logs, replay, role-based access, or managed support.
- Buyers who prefer commercial SLAs over a personal open-source project.
