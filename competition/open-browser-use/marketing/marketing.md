# Open Browser Use Marketing

## Audience

- Developers using Codex, Claude Code, Cursor, scripts, or MCP-capable agents who want access to real local Chrome.
- Agent builders who dislike hosted browser lock-in or need authenticated tabs, downloads, file choosers, clipboard, and CDP on the user's machine.
- Open-source users who prefer a small MIT-licensed bridge over a paid cloud browser service.

## Positioning

- Open Browser Use positions itself as platform-neutral Browser Use for local AI agents.
- The launch copy contrasts local real Chrome plus CLI/SDK/MCP with hosted automation services.
- It is not trying to look like a business workflow suite; it is trying to be the simplest installable browser handle for agent runtimes.

## Customer Model

- Open-source distribution through GitHub, npm, Homebrew, PyPI, Go modules, MCP directories, and skill package installation.
- No visible hosted pricing model in the current public surface.
- Growth depends on developer adoption, agent-runtime compatibility, and trust in the native-host/extension boundary.

## Captures

- Users who want their agent to operate logged-in local browser context.
- Developers who want browser control without learning a vendor-specific cloud API.
- Agent-runtime authors who need a neutral browser-control layer with multiple language SDKs.

## Misses

- Teams that need managed cloud sessions, concurrency, replay dashboards, support, and procurement.
- Nontechnical users who want templates, scheduling, and a visual builder.
- Security-conscious organizations that require formal policy controls before local agents touch real profiles.
