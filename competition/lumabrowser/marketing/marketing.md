# LumaBrowser Marketing

## Target customer

- AI builders who want MCP-native browser tools without writing glue code.
- QA engineers who already understand Selenium, Puppeteer, Playwright, CDP, flaky selectors, and network interception.
- Backend developers who prefer local APIs and webhooks over a browser-agent UI.

## Positioning

- "Programmable browser for AI agents" is the core frame.
- LumaBrowser sells protocol compatibility and selector resilience: REST, MCP, WebDriver, CDP, network watcher, and LLM fallback.
- It positions against brittle selectors and wrapper-heavy agent setups rather than against consumer AI browsers.

## Captures

- Developers who want to automate by API and only escalate to AI when selectors break.
- Teams that want local browser control without committing to a hosted browser vendor.
- QA/data-extraction users who need network capture and multiple client-library options.

## Does not capture

- Nontechnical users who need a polished agent workspace rather than an API product.
- Enterprises requiring central governance, seat management, procurement, and compliance.
- Users who need first-class visual run history, annotations, and approval controls.
