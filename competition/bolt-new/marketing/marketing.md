# Bolt.new Marketing

## Audience

- Developers, designers, PMs, founders, and no-code builders who want a working full-stack web app without local setup.
- JavaScript ecosystem users who value StackBlitz, browser IDEs, instant previews, and GitHub import/export.
- Builders who want more runtime control than v0-style static generation but less setup than local Cursor/Codex workflows.

## Positioning

- Bolt.new positions itself as "prompt, run, edit, and deploy" for full-stack web applications in the browser.
- The central claim is environment control: the AI can install packages, run Node servers, use the terminal, inspect browser output, and deploy.
- StackBlitz frames Bolt as useful for experienced developers, PMs, and designers, not only non-coders.

## Customer Model

- Bolt is free to start, with paid plans for more AI tokens and private projects.
- Token allowances govern AI usage; project size and repeated context syncing can materially increase token consumption.
- Team sharing, GitHub integration, StackBlitz continuity, Bolt hosting, custom domains, and Netlify publishing create expansion paths.

## Who They Capture

- Users who want the magic of prompt-to-app but still need real npm packages, dev servers, and code execution.
- Builders with existing public GitHub projects who want an AI agent to work inside a browser-hosted environment.
- Teams that want shareable browser workspaces before committing to a local development setup.

## Who They Miss

- Users building non-JavaScript stacks, native mobile apps, or workflows that cannot fit WebContainer constraints.
- Buyers who need predictable fixed-cost debugging.
- Security-conscious teams that require explicit review gates before an agent can repeatedly mutate runtime state.
