# Playwright MCP Marketing

## Audience

- Developers and AI coding-assistant users who need browser automation inside VS Code, Cursor, Claude Desktop, Windsurf, Claude Code, and similar clients.
- Test authors who want an assistant to inspect a real browser instead of hallucinating selectors from static docs or screenshots.
- Agent builders who need a free local browser-control primitive before adopting a hosted browser platform.

## Positioning

- Core claim: LLMs can interact with pages through structured accessibility snapshots without screenshots or visually tuned models.
- Market frame: existing coding agents need safe, deterministic browser primitives to navigate, inspect, and act in live apps.
- Product promise: official Playwright-backed browser control with the same ecosystem credibility as Playwright Test and Playwright Library.

## Customer Model

- Open-source, npm-distributed, no direct product pricing.
- Captures mindshare by becoming the default browser tool in AI coding workflows.
- Expands Playwright's role from testing framework to AI-agent browser substrate.

## Who They Capture

- Developers who already trust Playwright and want immediate browser automation.
- Teams using MCP-first assistants that can install a server quickly.
- Test-generation workflows where DOM inspection matters more than a managed browser fleet.

## Who They Miss

- Teams needing hosted browser concurrency, anti-bot bypass, session replay, billing controls, and managed proxies.
- Product teams needing opinionated agent traces, approval UX, skill routing, and workspace-specific browser state.
- Non-technical end users who do not configure MCP servers or npm tools.
