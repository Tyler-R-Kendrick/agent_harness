# Midscene.js Marketing

## Audience

- QA engineers and frontend teams dealing with high-churn UI automation.
- Agent builders who want UI operation tools that can work beyond DOM-backed web pages.
- Coding-tool users who want MCP-accessible UI actions and agent skills.

## Positioning

- Core claim: vision-driven UI automation for every platform.
- Market frame: DOM selectors and accessibility snapshots are not enough for canvas, mobile, desktop, or frequently changing interfaces.
- Product promise: describe automation in natural language through JavaScript or YAML, run it with Playwright/Puppeteer/bridge mode, debug it with replay reports, and expose it through MCP.

## Customer Model

- MIT-licensed open-source project with GitHub/community adoption.
- Adoption path is developer-led: docs, examples, Chrome extension, playgrounds, and package installation.
- Users supply model access and decide whether to run open-source or hosted vision models.

## Who They Capture

- Teams with brittle UI locators or high visual churn.
- Cross-platform automation teams that do not want separate stacks for web, mobile, and desktop.
- AI-tool developers looking for MCP-accessible UI control primitives.

## Who They Miss

- Teams with stable test IDs and mature deterministic test automation.
- Buyers who want a managed hosted-browser platform with account governance and billing controls.
- End users who want an opinionated browser workspace rather than a developer automation toolkit.

## Competitive Read

Midscene is a strong design reference for `agent-browser` because it treats replay reports, playgrounds, Chrome extension access, and MCP as product surfaces instead of afterthoughts. The counter-position is to make browser-specific state, approvals, cost, and evidence more precise than a broad cross-platform UI automation toolkit can.
