# OpenAI Codex Marketing

## Target Customer

- Professional developers and software teams already using ChatGPT, GitHub, VS Code-like editors, or OpenAI models.
- Engineering managers who want several coding agents running in parallel with central review.
- Business, Edu, and Enterprise admins who need workspace controls over local/cloud agent use.
- Increasingly, knowledge workers who use code-like workflows for analysis, documents, spreadsheets, dashboards, and internal automation.

## Positioning

- "Coding agent" and "command center for agents" rather than autocomplete.
- OpenAI frames Codex as a full lifecycle assistant: designing, building, shipping, maintaining, code review, and long-running delegated work.
- The platform story is cross-surface: ChatGPT, desktop app, CLI, IDE extension, cloud tasks, and GitHub review all share the same account and usage model.

## Customer Model

- Included in ChatGPT Free, Go, Plus, Pro, Business, Edu, and Enterprise plans, with limits and credits varying by plan.
- Paid users can encounter token-based credit metering, credit purchase options, and admin-controlled workspace settings.
- Enterprise buyers care about RBAC, local versus cloud execution controls, GitHub connection policy, and auditability.

## Who They Capture

- Developers who want model-owner convenience and are already in the OpenAI ecosystem.
- Teams that treat coding agents as a work queue: assign tasks, inspect diffs, request revisions, and open PRs.
- Users who value broad model capability and fast adoption more than local-first data boundaries.

## Who They Miss

- Teams that need page-level browser evidence, local state review, or deterministic UI traces as the primary artifact.
- Buyers who cannot route source code or task execution through model-owner cloud surfaces.
- Users who are sensitive to credit burn, opaque retries, or task cost variance.

## Agent Browser Opportunity

- Compete on evidence quality: screenshots, recordings, browser traces, approvals, and exact page state after each action.
- Make cost and authority visible while the work runs, not only after the diff is ready.
- Treat the browser as a first-class execution surface instead of a supporting implementation detail behind a coding task.
