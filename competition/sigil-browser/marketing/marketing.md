# Sigil Browser Marketing

## Audience

- Enterprise developers and security teams adopting browser agents inside existing Chrome profiles.
- Regulated or SSO-heavy organizations that cannot move browser work into a remote sandbox casually.
- AI-tool users who want Claude Code, Codex, Cursor, GitHub Copilot, or other MCP-capable assistants to use authenticated browser state.

## Positioning

- Core claim: agents can browse the user's real Chrome securely.
- Differentiator: deterministic guardrails are enforced outside the prompt, so policy is not just an instruction to the model.
- Efficiency claim: semantic page snapshots reduce token use relative to full DOM or screenshot strategies.
- Trust claim: no cloud sandbox or shared browser fleet is required for company browser data.

## Customer Model

- Free beta with a demo path.
- The page signals an enterprise sales motion around policy, audit, and regulated-industry deployment.
- Likely monetization vectors include team seats, organization policies, managed audit retention, and enterprise support.

## Who They Capture

- Security-conscious teams that like real-profile browser automation but worry about unbounded extension or MCP access.
- Developers already using AI coding assistants who need browser authority for logged-in internal apps.
- Buyers who want local execution language rather than managed browser infrastructure.

## Who They Miss

- Teams that want cloud browser scale, proxy/CAPTCHA handling, and hosted session orchestration.
- Solo users who want transparent pricing before trying a workflow.
- Buyers who need a complete agent workbench with traces, recordings, task queues, and recovery UX rather than a secure browser-control layer.
