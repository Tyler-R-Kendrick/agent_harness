# LaVague Marketing

## Audience

- Developers building AI web agents for end-user process automation.
- QA engineers who already use BDD/Gherkin and want faster Selenium/Pytest generation.
- Open-source contributors interested in Large Action Models, RAG over web pages, and browser action datasets.

## Positioning

- Core claim: framework for agent builders.
- Market frame: agents should navigate SaaS, docs, and web apps by converting objectives into reliable browser actions.
- QA promise: automate test writing and maintenance by turning Gherkin into tests that can be integrated into an existing test base.

## Customer Model

- Open-source GitHub project with Discord/community support.
- Enterprise contact path for buyers that want process automation or QA help.
- Users bring their own model keys by default and must manage model cost.

## Who They Capture

- Python/Selenium teams that want agentic test-generation experiments.
- QA organizations with BDD artifacts and repetitive browser test authoring.
- Researchers and open-source builders who want a hackable action-model framework.

## Who They Miss

- TypeScript-first teams that want a Playwright-native package as the default surface.
- End users who want a visible browser assistant instead of generated code.
- Enterprises that require default-off telemetry, formal audit trails, and deterministic approval gates before agent actions.

## Competitive Read

LaVague is less of a direct browser workspace competitor and more of a workflow competitor. It captures teams that start from BDD/test assets and want generated automation. `agent-browser` can learn from the Gherkin-to-run bridge while competing on live evidence, approvals, and local browser state.
