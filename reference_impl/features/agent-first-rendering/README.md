# Feature: Agent-First MCP Rendering

> Status: **Draft** | Created: 2026-04-04

## Problem Statement

Traditional browsers render web pages as-is and bolt AI assistance onto the side (sidebars, copilot panels). This forces users to context-switch between raw page content and AI-generated insights, and treats each tab as an isolated unit of interaction. The result: users manually synthesize information across tabs, manually navigate complex pages to find what matters, and get AI help that's disconnected from their actual task.

## Proposed Solution

Invert the rendering model. Instead of rendering pages by default and adding AI on top, the browser's agent parses pages in the active **working session** (workspace) and renders **MCP apps** as the primary interaction surface. Raw page rendering becomes a fallback or an embedded component within the MCP app — not the default.

## Core Concept

The agent sits between the network response and the user's screen. When a page (or set of pages) loads in a workspace, the agent:

1. **Parses** the page content (DOM, structured data, media, interactive elements)
2. **Understands** the user's current task context from the workspace
3. **Renders** an MCP app that surfaces the relevant content and interactivity

This operates across a **fidelity spectrum**:

| Fidelity Level | Description | Example |
|---|---|---|
| **Passthrough** | Raw page rendered as-is (traditional browser) | User explicitly opts out of agent rendering |
| **Extract** | Subset of page rendered as lightweight MCP app | YouTube player extracted, article text cleaned up |
| **Summarize** | Agent-generated summary with interactive drill-down | Research article → key findings + expandable sections |
| **Compose** | Custom MCP app synthesizing multiple tabs | Jira + Figma + PR → unified feature review interface |

## Key Abstractions

- **Working Session**: The set of active tabs/tab groups in a workspace (maps to the workspace concept)
- **Page Parse Pipeline**: Content extraction → structured representation → agent analysis
- **MCP App**: The rendering primitive — a self-contained interactive surface generated by the agent
- **Fidelity Negotiation**: The process by which the agent proposes a rendering tier and the user can adjust
- **Composition Engine**: The subsystem that merges multiple parsed pages into a single MCP app

## Relationship to Workspace Model

This feature builds directly on the workspace tree model. Each workspace node provides:

- **Context boundary**: The agent knows which tabs belong to the same task
- **Memory tier integration**: Hot workspaces get real-time MCP app updates; warm/cool workspaces get cached MCP app snapshots
- **Process isolation**: Each workspace's agent rendering pipeline runs in its own subprocess, isolated from others

## Success Criteria

1. User can open a set of tabs and receive a composed MCP app without manual configuration
2. Agent-rendered views load within acceptable latency bounds (see ADR-003)
3. Users can drill down from any MCP app view to the raw page
4. MCP apps can perform actions back on source pages (submit forms, approve PRs, move tickets)
5. The system degrades gracefully — if the agent can't parse a page, passthrough rendering kicks in
